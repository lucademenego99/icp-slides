Polymorphism

  • Polymorphism (in general): the ability to assume different forms
  • Polymorphism (in programming languages): the ability of a syntactic element to refer to elements of different types
  • Liskov's substitution principle: if X is a subtype of T, then variables of type T in a program can be substituted for variables of type X without altering any desirable properties of the program

Polymorphism in Java

  • A variable of reference type T may refer to any object whose type is T or its subtype.
  • Similarly, a formal parameter of reference type T may refer to actual parameters whose type is T or one of its subtypes.

What is the purpose of this?

  • Inheritance and polymorphism are fundamental in object-orientation
  • Together, they make it possible to:
    • keep the definition of a class "open" to subsequent extensions (specializations), without having to modify the code that uses it;
    • choose the most appropriate "view" to manage an object (general vs. specific) each time.
  • Strong positive impact on readability and maintainability of the code!

Dynamic binding: example

Static type and dynamic type

  • In the presence of polymorphism, a distinction is made between the static type (declared at compilation time) and the dynamic type (assumed at runtime) of a formal variable or parameter.
    • The substitution rule forces the dynamic type to be a subtype of the static type.
    • In Java, in an invocation x.f(x1, ..., xn), the implementation chosen for the method f depends also on the dynamic type of x and not only on its static type

Rule for dynamic binding

  • Let's assume to have:
    C o = ...;
o.m(...);
    the method chosen depends on the dynamic type of o, and is decided (at runtime) using the following logic:
    • We search inside the class C (static type of o) for the method m with the signature that is closest to the invocation.
      • The signatures to consider are those fixed at compile time, i.e. looking at the static type of the current parameters
    • We look at the dynamic type D of o; if it is a subtype of C, we have to check if it redefines (overrides) m. If yes, we use D's implementaion, otherwise C's.

The static type determines which methods can be invoked; the dynamic type determines which (re)definitions to perform.

Static and dynamic binding

  • C++ offers the programmer complex mechanisms for deciding whether to use dynamic binding (type decided at runtime) or static binding (type decided at compile time).
  • In Java, decisions are always made at runtime
    • ... except when it is possible to decide automatically at compile time, namely for:
      • constructors
      • private, static and final methods

Classes and methods final

  • It is possible to prevent the creation of subclasses of a certain class by defining the class final
  • Example:
    final class C { ... }
class C1 extends C {} // ERROR
  • Similarly, it is possible to prevent overriding of a method by defining it as final
  • Example:
    class C { final void f() { ... } }
class C1 extends C {
  void f() { ... } // ERROR
}

Useful when you want to prevent changing the behaviour of all or part of the class.

Forced conversions between reference types

Casting

  • You can force a conversion from a reference type T to a subtype T1 as long as...
    • ... the dynamic type of the converted object is a subtype of T1
      Object o = new Person();
Person a = o;           // ERROR, Object is not a subtype of Person
Person a = (Person) o;  // CORRECT (casting)
  • The implicit conversion allowed by polymorphism is called an upcast(ing).
  • The explicit conversion is called a downcast(ing).

Simple inheritance problems

Multiple inheritance: issues

  • In languages that support multiple inheritance (e.g., C++) it is possible to inherit two or more methods with the same signature from multiple superclasses...
  • ... which creates a conflict between different implementations

Java Approach: interface

  • Distinguish between:
    • a simple hierarchy of inheritance (implementation)
    • a multiple hierarchy of specialization (types)
  • ... introducing the construct of interface
  • It allows to separate the use of inheritance in order to reuse code and the use in order to describe a hierarchy of types

Interfaces

  • An interface is a collection of method signatures.
    • It can be viewed as a class with no attributes, whose methods are all public and abstract
  • Syntax:
    interface <name> {
  <methods list : signatures only, no coding>
}

    • Since Java 8 it is possible to have "default methods" with a body... not considered here

  • An interface can also contain constants; however, in general this practice is not recommended
  • Sometimes you use the name alone (no methods) to "tag" classes with special properties (tagging interfaces).
    • E.g. Cloneable, Serializable, Remote, ...

Interfaces and inheritance

  • An interface can inherit from one or more interfaces:
    interface <Iname> extends <Iname_1>,...,<Iname_n> { ... }
    • The inheritance hierarchy between interfaces defines a hierarchy of types
  • A class can implement one or more interfaces
    • if the class is not abstract, it must provide an implementation for all methods present in the interfaces it implements; vice versa, the class is abstract
      class <Iname> implements <Iname_1>,...,<Iname_n> { ... }

Information hiding in Java

  • The visibility of attributes and methods of a C class can be:
    • public
      • visible to all
      • are inherited
    • protected
      • visible to the classes declared in the C package
      • visible to C subclasses, even if defined in another package
    • "package" (no modifier specified)
      • visible only to classes declared in the C package
    • private
      • visible only within the class in which they are defined
      • not visible in subclasses
